# OpenHarmony 文档编写规范（精简版）

> 仅保留高阶组件（atomicservice/advanced）API 参考文档检视相关规范。

---

## 一、文件命名

| 类型 | 格式 |
|------|------|
| 高级组件 | d.ts 同名，`.` → `-` |
| 原子化服务 | 同上 |
| 系统接口 | 公共名 + `-sys.md` |

---

## 二、文档结构规范

标准章节：`# 组件名称` → `> **说明：** 起始版本 + 原子化服务API` → `## 导入模块` → `## 子组件`（无则写"无"）→ `## 属性/事件`（不支持时注明）→ `## 构造函数/方法`（含 `**参数：**` 标题）→ `## 示例代码`（figures/截图）

必须项：
- `## 导入模块` — 必须
- `## 子组件` — 无则写"无"
- `**原子化服务API：**` — 构造函数自身及含原子化服务API的参数/方法上均需标注

---

## 三、d.ts 标签与文档字段映射

| d.ts 标签 | 文档字段 |
|-----------|----------|
| `@since` | `**起始版本**：x.x.x` 或上标 `<sup>x+</sup>` |
| `@atomicservice` | `**原子化服务API：** 从 API version x 开始支持` |
| `@systemapi` | `> **说明：** 本模块接口为系统接口。` |
| `@syscap` | `**系统能力**：SystemCapability.xxx` |
| `@permission` | `**需要权限**：ohos.permission.xxx` |
| `@StageModelOnly` | `**模型约束**：此接口仅可在 Stage 模型下使用。` |
| `@装饰器` | `**装饰器类型**：@装饰器名称` |

---

## 四、表格规范

- 表头风格全文统一（"名称" vs "参数名"）
- 同一列内容格式一致，句末标点统一
- 不出现空白单元格
- 列头：新版"名称\|类型\|必填\|装饰器类型\|说明"或旧版"名称\|类型\|只读\|可选\|说明"

---

## 五、图片规范

- 统一存放在同级 `figures/` 目录
- 命名：字母、数字、中划线，语义化
- 涉及 UI 的示例**必须**提供效果示意图

---

## 六、代码块规范

- 添加语言标记（`ts`/`js`）
- ArkTS 示例在文件头添加 `// xxx.ets` 注释
- 注释统一**不带开头空格**（`//获取` 而非 `// 获取`）
- 代码逻辑和语法正确，**可直接编译运行**

---

## 七、版本标注规范

两段式（API 26.0.0 之前）：`> **说明：** 本模块首批接口从 API version x 开始支持。` 新接口用 `<sup>x+</sup>` 必须闭合。

三段式（API 26.0.0 起）：`**起始版本**：26.0.0`

> 错误示例：`onReceive<sup>20+<sup>` → 正确：`onReceive<sup>20+</sup>`

---

## 八、链接规范

- 使用 Markdown 链接语法，禁止 `<a>` 标签
- 锚点指向具体"对象说明"段落，**而非章节标题**
  - ❌ `[SymbolGlyphModifier](ts-universal-attributes-attribute-modifier.md#自定义modifier)`
  - ✅ `[SymbolGlyphModifier](ts-universal-attributes-attribute-modifier.md)`

**错误码链接格式：**
- ❌ `以下错误码的详细介绍请参见[页面路由错误码](../errorcode-router.md)。`
- ✅ `以下错误码的详细介绍请参见[ohos.router(页面路由)](../errorcode-router.md)错误码。`

---

## 九、提示语规范

| 标识 | 用途 |
|------|------|
| **说明** | 帮助提示、参考信息 |
| **注意** | 可能导致异常 |
| **警告** | 不可恢复异常 |

---

## 十、系统接口分离

| 类型 | 文件名 | 标题 |
|------|--------|------|
| 公共接口 | `xxx.md` | `# 模块名` |
| 系统接口 | `xxx-sys.md` | `# 模块名（系统接口）` + `> **说明：** 本模块接口为系统接口。` |

---

## 十一、示例代码规范

| 要求 | 说明 |
|------|------|
| 可编译 | 放入 IDE 可正常编译 |
| 可运行 | 运行后效果和文档一致 |
| 截图证明 | UI 组件必须提供效果图 |

**回调/Promise 句式：**
- `callback<boolean>`：返回 true 表示 xxx；返回 false 表示 xxx
- `AsyncCallback<void>`：操作成功 err 为 undefined，否则为错误对象
- `Promise<void>`：无返回结果的 Promise 对象

---

## 十二、参数描述要求

必须阐明：含义与用途、场景、选取建议、参数间关联。

取值说明必须包括：取值范围、单位、默认值、取值原则/建议值、固定格式样例。

---

## 十三、API 规格加固要求

| 类型 | 检查项 |
|------|--------|
| String | 长度范围、有效集合、特殊格式（邮箱/URL/JSON等） |
| Number | 边界值 |
| Object | 子参数规格 |
| Bool | true/false 各为何种场景 |
| — | 默认值正确 |

---

## 十四、约束限制完整性

| 要求 | 说明 |
|------|------|
| 权限说明 | 必要、无超出、无遗漏 |
| 设备品类差异 | 不同设备的支持/效果差异描述正确 |

---

## 十五、错误码规范

| 要求 | 说明 |
|------|------|
| 可执行 | 按步骤能正确解决问题 |
| 步骤明确 | 明确、具体、可执行 |

**模板要素：** 错误信息、错误描述、可能原因（按可能性排序）、处理步骤（含子步骤）
